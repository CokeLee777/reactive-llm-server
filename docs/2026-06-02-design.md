# reactive-llm-server 설계 문서

**작성일:** 2026-06-02  
**목표:** Spring-AI, OpenAI SDK 없이 순수 Java + Reactor로 LLM 스트리밍 서버 직접 구현  
**언어:** Java (주), Reactor (비동기 처리)  
**예상 기간:** 11-13주 (주당 15-20시간)

---

## 1. 프로젝트 목적

HTTP/Streaming(SSE, Chunked Encoding)과 Java Reactor(Flux/Mono)의 동작 원리를 추상화 라이브러리 없이 직접 구현하면서 깊게 이해한다.

ChatGPT처럼 AI 응답이 실시간으로 타이핑되는 효과를 **직접 만든 코드**로 구현하는 것이 최종 목표다.

## 2. 전체 아키텍처

```
LLM API (OpenAI / Claude / Gemini)
        │
        │  HTTP/1.1 Chunked Transfer Encoding
        │  SSE (data: {"delta": "..."} ... data: [DONE])
        ▼
┌─────────────────────────────┐
│   LLM HTTP Streaming Client  │  ← Phase 1
│   (순수 Java HttpClient)      │
│   java.net.http.HttpClient가  │
│   Chunked Encoding 자동 처리  │
└─────────────────────────────┘
        │
        │  InputStream (디코딩 완료된 SSE 텍스트)
        │  ※ blocking I/O → boundedElastic 스케줄러에서 실행
        ▼
┌─────────────────────────────┐
│   SSE Parser                 │  ← Phase 1
│   (라인 읽기, 이벤트 파싱,    │
│    [DONE] 종료 감지)          │
└─────────────────────────────┘
        │
        │  Reactor Flux<String> (토큰 스트림)
        ▼
┌─────────────────────────────┐
│   SSE Server                 │  ← Phase 2
│   (Reactor Netty 기반)        │
└─────────────────────────────┘
        │
        │  SSE (text/event-stream)
        ▼
┌─────────────────────────────┐
│   Browser / Client           │  ← Phase 3
│   (토큰이 실시간 타이핑 효과)  │
└─────────────────────────────┘
```

### 패키지 의존성 방향

```
Main.java (조합)
    ├── config/    (의존성 없음)
    ├── domain/    (의존성 없음)
    ├── sse/       (의존성 없음)
    ├── client/    → domain/, sse/
    ├── server/    (의존성 없음)
    └── chat/      → domain/, client/
```

> 의존성은 단방향이다. 하위 레이어가 상위 레이어를 참조하지 않는다.  
> `Main.java`에서 모든 컴포넌트를 생성하고 `BiFunction` 람다로 조합하므로 `server/ → chat/` 의존성이 생기지 않는다.

### HTTP 요청 처리 흐름

```
클라이언트 요청
        │
        ▼
  SseServer (라우팅)
        │
        ├─ /chat POST ──→ SseConnectionHandler
        │                    │ 1. Content-Type: text/event-stream 헤더 설정
        │                    │ 2. 주입된 BiFunction 호출
        │                    │       └─ ChatEndpoint.handle(request, response)
        │                    │              ├─ X-Conversation-Id 응답 헤더 설정
        │                    │              └─ Flux<String> 반환
        │                    │ 3. SseEmitter.emit(flux) → "data: ...\n\n" 포맷 변환
        │                    │ 4. HttpServerResponse.sendString(formattedFlux)
        │                    ▼
        │               HttpServerResponse (Netty)
        │
        └─ GET /* ──────→ StaticFileHandler
```

> `SseConnectionHandler`가 요청 처리 전체를 조율한다.  
> `ChatEndpoint`는 `HttpServerResponse`를 직접 받아 `X-Conversation-Id` 헤더를 설정한 뒤 `Flux<String>`을 반환한다.  
> HTTP 헤더는 body 전송 전에 설정해야 하므로, `ChatEndpoint`가 헤더를 설정하는 시점(2단계)은 SseEmitter 호출(3단계) 이전이다.

## 3. 사전 학습 (Week 0)

Phase 1 시작 전에 Reactor 기초를 먼저 익혀야 한다. Phase 1에서 `Flux`를 직접 사용하는데, 개념 없이 시작하면 구현보다 디버깅에 더 많은 시간을 쓴다.

**학습 내용:**

| 주제 | 핵심 개념 |
|------|-----------|
| Cold vs Hot Publisher | subscribe 시점에 데이터를 생성하는가, 아니면 이미 흐르고 있는가 |
| Flux 생성 패턴 | `Flux.create()`, `Flux.generate()`, `Flux.fromIterable()` |
| 기본 연산자 | `map()`, `flatMap()`, `filter()`, `take()` |
| Subscription 생명주기 | `subscribe()` → `onNext()` → `onComplete()` / `onError()` |
| CompletableFuture 연동 | `Mono.fromFuture(() -> ...)` — supplier 형태로 lazy 실행 보장 |
| Blocking I/O 처리 | `subscribeOn(Schedulers.boundedElastic())` — blocking 코드를 별도 스레드 풀에서 실행 |
| StepVerifier | Reactor 비동기 스트림 테스트 도구 |

## 4. Phase별 상세 설계

### Phase 1 - LLM API와 직접 HTTP 통신 (3-4주)

**목표:** SDK 없이 LLM API의 SSE 응답을 파싱하고 Reactor Flux로 변환한다.

> **중요:** `java.net.http.HttpClient`는 `Transfer-Encoding: chunked`를 **자동으로 처리**한다.
> 직접 파싱할 대상은 청크 인코딩이 아니라 SSE 포맷(`data: {...}\n\n`)이다.

**학습 핵심:**
- SSE 응답 형식(`data: {...}\n\n`)을 직접 파싱하는 방법
- `data: [DONE]`을 감지해서 `Flux`를 정상 종료(`onComplete`)시키는 방법
- `HttpClient.sendAsync()`의 `CompletableFuture`를 `Mono.fromFuture(() -> ...)`로 lazy하게 Reactor에 연결하는 방법
- `InputStream`의 blocking 읽기를 `subscribeOn(Schedulers.boundedElastic())`으로 안전하게 처리하는 방법
- 환경변수로 API Key를 안전하게 관리하는 방법

**구현 목록:**

| 컴포넌트 | 패키지 | 설명 |
|---------|--------|------|
| `Message` | `domain/` | 대화 메시지: `record Message(String role, String content)`. 의존성 없음 |
| `AppConfig` | `config/` | 환경변수(`OPENAI_API_KEY` 등)에서 설정 로드 |
| `LlmClient` | `client/` | 인터페이스: `Flux<String> stream(List<Message> history)` |
| `OpenAiLlmClient` | `client/` | `LlmClient` 구현체. `HttpClient`로 POST 요청 후 `TokenFluxFactory.fromStream()`으로 `Flux<String>` 생성 |
| `SseLineReader` | `sse/` | `InputStream`에서 한 줄씩 읽어 `Flux<String>` (raw SSE 라인) 반환. `boundedElastic`에서 실행 |
| `SseEventParser` | `sse/` | `data: {...}` 파싱 + 토큰 추출. `data: [DONE]` 감지 시 스트림 종료 신호 반환 |
| `TokenFluxFactory` | `sse/` | `fromStream(InputStream): Flux<String>` — `SseLineReader` + `SseEventParser` 조합으로 토큰 스트림 생성 |

**배우는 것:**
- SSE 스펙(`data:`, `event:`, `id:`, `retry:` 필드)
- `HttpClient.sendAsync()` + `BodyHandlers.ofInputStream()` 조합
- `Mono.fromFuture(() -> ...)` — supplier 형태를 써야 lazy 실행. 이미 생성된 Future를 넘기면 subscribe 전에 HTTP 요청이 즉시 발송됨(eager)
- `Flux.create()` + `subscribeOn(Schedulers.boundedElastic())` — blocking I/O를 Flux로 감싸는 패턴
- `[DONE]` 시그널 처리 — 스트림 종료 조건을 직접 구현하는 방법
- `StepVerifier`로 `Flux<String>` 단위 테스트 작성

**에러 처리 전략:**

| 상황 | 처리 방법 |
|------|-----------|
| 네트워크 타임아웃 | `HttpClient`의 `timeout()` 설정 + `onErrorResume()` |
| API 4xx (인증 실패, rate limit) | `onErrorMap()`으로 도메인 예외로 변환 후 상위 전달 |
| API 5xx | `retryWhen(Retry.backoff(3, Duration.ofSeconds(1)))` — 지수 백오프 재시도 |

**Phase 1 테스트 전략:**

단위 테스트에서 실제 OpenAI API를 호출하지 않는다. `ByteArrayInputStream`으로 가짜 SSE 텍스트를 주입해서 테스트한다.

```java
String fakeSse = "data: {\"choices\":[{\"delta\":{\"content\":\"Hello\"}}]}\n\n"
               + "data: [DONE]\n\n";
InputStream input = new ByteArrayInputStream(fakeSse.getBytes(StandardCharsets.UTF_8));

StepVerifier.create(tokenFluxFactory.fromStream(input))
    .expectNext("Hello")
    .expectComplete()
    .verify();
```

### Phase 2 - SSE 서버 직접 구현 (3-4주)

**목표:** Spring WebFlux 없이 Reactor Netty만으로 SSE 서버를 구현하고 Reactor의 핵심 개념을 깊게 이해한다.

**학습 핵심:**
- Publisher/Subscriber 패턴이 실제로 어떻게 동작하는가
- Backpressure가 무엇이고 왜 필요한가
- SSE 서버가 클라이언트에게 이벤트를 어떻게 전달하는가

**구현 목록:**

| 컴포넌트 | 설명 |
|---------|------|
| `SseServer` | Reactor Netty 기반 HTTP 서버. `BiFunction` 핸들러를 주입받아 라우팅 |
| `SseConnectionHandler` | 요청 조율자: 헤더 설정 → BiFunction 호출 → SseEmitter 변환 → sendString |
| `SseEmitter` | `emit(Flux<String>): Flux<String>` — `data: ...\n\n` 포맷으로 변환만 담당 |
| `BackpressureHandler` | `SseEmitter` 내부 유틸. `BackpressureStrategy` enum(DROP/BUFFER/ERROR)을 생성자로 주입받아 해당 전략 적용 |

> **역할 경계:**
> - `SseConnectionHandler` — 전체 요청 조율. `BiFunction<HttpServerRequest, HttpServerResponse, Flux<String>>` 호출
> - `SseEmitter` — 포맷 변환만. `Flux<String>` → `Flux<String>` (`data: token\n\n` 형식)
> - `BackpressureHandler` — `SseEmitter` 내부 전략 유틸. `BackpressureStrategy.DROP/BUFFER/ERROR`를 생성자로 주입해 전략 교체 가능
> - 실제 바이트 전송은 `HttpServerResponse.sendString(formattedFlux)`

> **`SseServer` 조합 방법 (Main.java):**
> Phase 2에서는 람다로 임시 Flux를 생성해 단독 테스트한다.
> Phase 3 통합 시 `Main.java`에서 아래와 같이 조합한다.
> ```java
> // Main.java (Phase 3 통합 시)
> SseConnectionHandler sseHandler = new SseConnectionHandler(
>     (request, response) -> chatEndpoint.handle(request, response),
>     sseEmitter
> );
> SseServer server = new SseServer(sseHandler, staticFileHandler);
> ```

**배우는 것:**
- `Content-Type: text/event-stream` 응답의 동작 방식
- Reactor의 `Subscriber.request(n)` — Backpressure 작동 원리
- `onBackpressureDrop()` / `onBackpressureBuffer()` / `onBackpressureError()` — 세 전략의 차이
- `onErrorResume()`, `retryWhen()` — 에러 핸들링 패턴

**Phase 2 테스트 전략:**

```java
// 1. SSE 포맷 변환 검증 — 핵심 기능: "data: ...\n\n" 포맷이 맞는지 확인
Flux<String> tokenFlux = Flux.just("Hello", "world");

StepVerifier.create(sseEmitter.emit(tokenFlux))
    .expectNext("data: Hello\n\n")
    .expectNext("data: world\n\n")
    .expectComplete()
    .verify();

// 2. Backpressure 취소 테스트 — Subscriber가 중간에 취소해도 문제없는지 확인
Flux<String> tokenFlux = Flux.just("Hello", " ", "world");

StepVerifier.create(sseEmitter.emit(tokenFlux), 1)
    .thenRequest(1)
    .expectNextCount(1)
    .thenCancel()
    .verify();
```

### Phase 3 - 실용적인 기능 추가 (4-5주)

**목표:** Phase 1 + Phase 2를 조합해서 실제로 사용 가능한 LLM 스트리밍 서버를 완성한다.

**구현 목록:**

| 컴포넌트 | 패키지 | 설명 |
|---------|--------|------|
| `ConversationManager` | `chat/` | 대화 히스토리 관리 (in-memory, `ConcurrentHashMap` 사용) |
| `LlmProviderRouter` | `chat/` | 요청 파라미터(`provider=openai` 등)를 읽어 `Map<String, LlmClient>`에서 구현체 반환 |
| `ChatEndpoint` | `chat/` | `handle(HttpServerRequest, HttpServerResponse): Flux<String>` |
| `StaticFileHandler` | `server/` | Reactor Netty로 정적 파일 직접 서빙 |
| Web UI | `resources/static/` | 토큰이 실시간으로 타이핑되는 효과의 간단한 HTML/JS 페이지 |

> **`ConversationManager` Thread Safety:**
> `ConcurrentHashMap<String, List<Message>>`로 대화 ID별 히스토리를 관리한다.
> 히스토리 초기화는 `computeIfAbsent()`로 원자적으로 수행한다. `map.get()` → `map.put()` 패턴은 두 스레드가 동시에 같은 ID로 진입할 때 Race Condition이 생기므로 사용하지 않는다.
> `List<Message>`는 `Collections.synchronizedList()`로 감싸서 개별 append 연산의 동시 쓰기를 보호한다.
>
> ```java
> // 올바른 패턴
> List<Message> history = map.computeIfAbsent(id,
>     k -> Collections.synchronizedList(new ArrayList<>()));
> history.add(message);
> ```
>
> **메모리 제한:** in-memory 저장이므로 대화 히스토리가 무한히 증가한다. 장시간 운영 시 OOM 위험이 있으며, TTL 기반 만료 기능은 이 프로젝트 범위 밖이다.
>
> **대화 흐름:**
> - 신규 대화: 클라이언트가 `X-Conversation-Id` 헤더 없이 요청 → `ChatEndpoint`가 `UUID.randomUUID()`로 생성 후 `HttpServerResponse`에 `X-Conversation-Id` 헤더 설정
> - 기존 대화 이어가기: 클라이언트가 `X-Conversation-Id` 헤더에 ID 포함 → `ConversationManager`에서 히스토리 조회 후 LLM 요청에 전달

> **`LlmProviderRouter`:**
> 요청 파라미터 `provider`(예: `openai`, `claude`)를 읽어 `Map<String, LlmClient>`에서 해당 구현체를 꺼낸다.
> `Main.java`에서 Map을 조립해 주입하므로 새 Provider 추가 시 `LlmProviderRouter` 수정 없이 Map에 항목만 추가하면 된다.

## 5. 기술 스택

| 분류 | 기술 | 이유 |
|-----|------|------|
| 언어 | Java 25 | 최신 LTS, record/sealed class 등 현대 문법 활용 |
| 비동기 | Project Reactor 3.x | Flux/Mono 내부 동작 학습 목적 |
| 네트워크 | Reactor Netty | Spring WebFlux 없이 낮은 레벨 접근 |
| HTTP Client | `java.net.http.HttpClient` | SDK 없이 순수 HTTP 통신 (Chunked Encoding 자동 처리) |
| 빌드 | Gradle | 의존성 관리 |
| 테스트 | `reactor-test` (StepVerifier) | Reactor 비동기 스트림 검증 |
| LLM API | OpenAI API (기본), Claude API (선택) | SSE 스트리밍 지원 |

## 8. 성공 기준

- [ ] SDK 없이 OpenAI API의 스트리밍 응답에서 SSE 포맷을 직접 파싱할 수 있다
- [ ] `data: [DONE]` 시그널을 감지해서 `Flux`를 정상 종료시킬 수 있다
- [ ] `Mono.fromFuture(() -> ...)`로 CompletableFuture를 lazy하게 Reactor 체인에 연결할 수 있다
- [ ] blocking `InputStream` 읽기를 `Schedulers.boundedElastic()`으로 안전하게 처리할 수 있다
- [ ] `ByteArrayInputStream`으로 실제 API 없이 파서를 단위 테스트할 수 있다
- [ ] `Flux<String>`으로 토큰 스트림을 표현하고 `StepVerifier`로 검증할 수 있다
- [ ] Reactor Netty로 SSE 서버를 직접 구현할 수 있다
- [ ] Backpressure 세 가지 전략(drop/buffer/error)을 각각 구현하고 차이를 설명할 수 있다
- [ ] `LlmClient` 인터페이스를 활용해 새로운 LLM Provider를 추가할 수 있다
- [ ] 브라우저에서 토큰이 실시간으로 타이핑되는 것을 확인할 수 있다
- [ ] 코드를 보면서 HTTP/Streaming과 Reactor의 동작 원리를 설명할 수 있다
